You are working in a dotfiles repository. Your job is to take all uncommitted and untracked changes and split them into logical, small commits — then write a shell script that performs those commits.

## Repository structure

This is a dotfiles repo organized into `packages/`. Each package is a tool or program:
- `packages/nvim/` — Neovim config
- `packages/fish/` — Fish shell (config.fish, conf.d/, functions/)
- `packages/ghostty/` — Ghostty terminal
- `packages/shell-common/` — Shared shell config (aliases, environment)
- `packages/_home/` — Home directory dotfiles (bashrc, bash_profile, etc.)
- `packages/scripts/` — Custom scripts
- `packages/sketchybar/` — macOS bar
- `packages/aerospace/` — Aerospace window manager
- Other packages follow the same pattern.
- Top-level files: `.gitignore`, `Justfile`, `dotfiles.py`

## Commit message style

Follow this exact style:
- Format: `<scope>: <short description>`
- Scope = the package name or top-level file name (e.g., `nvim`, `fish`, `ghostty`, `shell`, `bash`, `Justfile`, `gitignore`)
- Description is lowercase, no period, imperative mood
- Keep it short and casual — no need to be overly formal
- For lock file updates: just `nvim: lazy-lock`
- For new packages: `<package>: init`
- When a commit touches multiple related shells (bash + fish), combine: `bash/fish: <description>`

Examples from this repo's history:
- `nvim: update lsp config`
- `nvim: fidget: fix deprecated nvimtree integration`
- `ghostty: init`
- `fish: add docker completions`
- `shell: load rustup env`
- `bash: add local bin to path`
- `gitignore pycache`
- `Justfile: fix aerospace build path`
- `nvim: lazy-lock`
- `fish: fix error when go is not installed`

## Grouping rules

1. **One package per commit** — don't mix changes across unrelated packages.
2. **Separate config changes from lock file updates** — e.g., nvim plugin config changes and lazy-lock.json should be separate commits.
3. **Group related changes within a package** — e.g., if you switch a colorscheme in nvim and also add the colorscheme plugin, that's one commit.
4. **Cross-package commits are OK when logically coupled** — e.g., adding LM Studio to PATH in both bashrc and fish config is one commit.
5. **New untracked packages get an "init" commit** — e.g., `sketchybar: init`.
6. **Small, obvious changes can stand alone** — a one-line alias addition is its own commit.
7. **Order commits sensibly** — dependencies first (e.g., add a colorscheme file before the config that references it).

## Output

Output ONLY a bash script (no markdown fences, no explanation) that:
- Starts with `#!/bin/bash` and `set -e`
- For each logical commit: `git add <files>` then `git commit -m "<message>"`
- Add a short comment before each group explaining what it does
- Does NOT push to remote

The current repo state (git status, git diff, untracked files) is appended below.
