You are working in a dotfiles repository. Your job is to take all uncommitted and untracked changes and split them into logical commits, then write a shell script that performs those commits.

## Commit message style

Follow this exact style:
- Format: `<scope>: <short description>`
- Scope should describe the part of the repo or change area, or use a top-level file name when that is the clearest description
- Description is lowercase, no period, imperative mood
- Keep it short and casual, but specific enough to reflect the actual change
- For tightly related changes that belong together, use one commit
- For supporting metadata or generated files that are part of the same change, keep them with the parent change
- Do not copy example wording; infer the real subject from the diff itself

## Grouping rules

1. **Group by shared intent** - keep changes together when they are part of the same user-visible or maintenance task.
2. **Split only when it improves clarity** - separate commits only when the changes are independent enough to review, revert, or reason about separately.
3. **Keep supporting changes with their cause** - include generated files, metadata, and follow-on adjustments in the same commit when they exist because of the main change.
4. **Do not split by directory alone** - file location should not force a commit boundary if the changes belong together.
5. **Prefer the real subject over placeholders** - derive commit subjects from the actual diff, not from generic examples or directory names.
6. **Order commits sensibly** - dependencies first, then the changes that rely on them.

## Output

Output ONLY the bash script — no markdown fences, no explanation, no preamble, no commentary before or after. The entire response must be the script and nothing else.

Only commit files that were included in the prompt. If an untracked package tree was not shown in the prompt because it was excluded upstream, do not add it, do not mention it, and do not create commits for it.

The script must:
- Starts with `#!/bin/bash` and `set -e`
- For each logical commit: `git add <files>` then `git commit -m "<message>"`
- Add a short comment before each group explaining what it does
- Does NOT push to remote
- Does NOT add any Co-Authored-By or Signed-off-by trailers to commit messages
